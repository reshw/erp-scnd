## 마이너스통장 정산 기산일 (날짜기준 / 특정요일기준)

**결정일:** 2026-07-06
**구현 범위:** 정산 주기(월 1회) 산정 방식만 개선. 이자율 변동 이력은 미반영.

### 무엇을 만들었나

- `loans.settlement_type` ('date' | 'weekday') + `settlement_day` / `settlement_weekday` / `settlement_week_of_month` 컬럼 추가 (`012_overdraft_settlement_schedule.sql`)
- "특정요일기준"은 "매월 N번째 요일"(달력상 단순 카운트)로 기산일을 잡고, **기산일이 주말이면 부과일(전표 날짜)은 익영업일로 이월** (`src/lib/loans/overdraftSettlement.ts`)
  - 근거: 실사례 부과일 2026-04-13, 05-11, 06-15(모두 월요일)가 처음엔 "2번째 월요일 + 1일 겹치면 건너뛰기" 규칙처럼 보였으나, 은행 앱의 다음 납입일이 2026-07-12(일요일)로 확인됨 → 실제 규칙은 **"매월 2번째 일요일 기산 + 주말이라 익영업일(월) 부과"**. 4~7월 데이터 전부 일치
- **이자 계산 기간의 경계는 기산일 기준**: 전월 기산일 다음날 ~ 이번달 기산일 당일 포함. 부과일(익영업일)은 전표 날짜로만 사용
  - 한때 "부과일 경계"(전월 부과일+1 ~ 이번달 부과일)로 바꿨다가 되돌림: ERP 잔액이 실제보다 낮게 기록된 상태에서도 부과일 경계 계산(415,670)이 실제 청구(415,559)를 초과 → 잔액을 맞추면 더 커지므로 부과일 경계는 확정적으로 과대계산
  - 전표를 은행과 일치시킨 후 검증: 기산일 경계 + **금리 변동 익일 적용**(5/13 변경 → 5/14부터 신금리)으로 계산 415,565원 vs 실제 415,559원 — **6원 오차로 사실상 확정**. 잔여 오차는 은행의 구간별 원미만 절사 방식 차이로 추정, 전표 발행 시 수동 보정으로 충분하다고 판단 (완전 일치가 필요해지면 은행 앱의 구간별 이자계산 내역 확보 후 절사 로직 구현)

### 마통 프로젝트별 이자 분해 — 2026-07-06 추가

마통 하나(잔액 ~79M)를 여러 프로젝트가 나눠 쓰는 상황(일반/JH407/JH1307). 이자가 잔액에 선형이라 프로젝트별 분해가 수학적으로 깔끔함.

- **고정금액 트랜치(`loan_tranches`) 방식을 먼저 만들었다가 같은 날 폐기**: 인출 전표가 이미 프로젝트별로 귀속돼 있어서(전표 헤더의 project_id), 고정금액을 별도로 등록·관리할 필요가 없음. 전표 데이터에서 프로젝트별 일별잔액을 직접 뽑아 분해하는 게 이중 관리 없는 정답
- 이자 계산 API가 `journals.project_id` 기준으로 일별잔액을 프로젝트별 분해 → 프로젝트별 이자 + 전표 버튼 제공. 분해 합계는 전체 이자와 1원 단위 일치 (최대 항목에서 반올림 오차 조정)
- 검증: 2026-06 정산 415,565 = JH1307 259,163 + 일반 128,899 + JH407 27,503 ✓
- 이자 전표 템플릿: 이자비용(차변) / **잔액 추적 계정**(대변, 장기차입금+마통 거래처) — 마통 이자는 통장 인출이 아니라 대출잔액에 가산되는 실무 패턴 반영
- 전표는 프로젝트별 개별 발행 (전표 헤더에 project_id가 있는 구조라 한 전표에 여러 프로젝트 배분 불가 — 라인별 프로젝트는 스키마 변경이 커서 보류)
- 마이그레이션: 마통 대출 3개(일반/JH407/JH1307로 쪼개 등록했던 것) → 마스터 1개(project 없음, 전체 전표 집계)로 통합, 나머지 2개 삭제
  - 공휴일 이월은 미반영 (공휴일 테이블 필요) — 기산일이 공휴일과 겹치는 달에 부과일이 하루 어긋날 수 있음, 실제 발생 시 반영 검토
- `settlement_type='date'`이고 `settlement_day`가 비어있으면 기존 동작(매월 1일~말일 전체) 그대로 유지 — 기존 마통 데이터 하위호환

### 이자율 변동 이력 관리 — 당초 보류했다가 같은 날 구현

처음엔 "금리 변동일을 알 수가 없다"는 이유로 보류했으나, 당일 사용자가 변동일 확인(2026-05-13부터 5.55%) → 바로 구현.

- `loan_rate_history` 테이블 (`013_loan_rate_history.sql`): loan_id + effective_date + annual_rate. 변동일 당일부터 새 금리 적용, 첫 변동일 이전은 `loans.interest_rate`(계약 금리) 사용
- 일별 이자 계산(`/api/loans/[id]/overdraft-interest`)이 날짜별 유효 금리를 조회해 반영
- 마통 상세 페이지에 금리 변동 이력 추가/삭제 UI + 일별잔액 미리보기에 적용 연이율 컬럼
- 일반 대출(원리금균등 등)의 상환 스케줄에는 미반영 — 마통만 대상. 일반 대출 금리 변동이 실제로 발생하면 그때 `calcSchedule` 반영 검토

## 미결잔액 관리 (미지급금·선급금·선수금 등 임시계정)

**결정일:** 2026-05-18  
**구현 범위:** 거래처별 잔액 요약만 (Phase 1)

### 무엇을 만들었나

- `/clearings` 페이지: 계정과목 선택 → 거래처별 발생액/반제액/미결잔액 테이블
- 클릭 시 `/ledger` 로 드릴다운 (계정 + 거래처 필터 자동 적용)

### 무엇을 안 만들었나: 반제 매칭 (Phase 2)

`clearings` 테이블을 추가해서 발생 라인과 반제 라인을 1:1로 연결하는 기능.

#### 왜 보류했나

**자동매칭이 현실적으로 어렵다:**
- 발생일과 반제일이 다른 게 기본 (당연히 날짜 매칭 불가)
- 부분 지급 / 합산 지급 / 같은 금액 반복 발생 등 예외가 많음
- FIFO 자동제안 수준은 가능하지만 오매칭 위험
- 적요 텍스트 AI 분석은 적요를 꼼꼼히 쓰지 않으면 useless

**과했던 것:**
- `clearings` 테이블 마이그레이션
- 라인별 "미결/부분반제/완결" 상태 UI
- 자동매칭 + 수동매칭 모달

#### 거래처별 잔액만으로 충분한 이유

미지급금 분석의 실질적 목적은 "거래처 A에 얼마 남아있나"이지,  
"#123 인보이스가 #167 지급건으로 상쇄됐는가"가 아닌 경우가 대부분.  
잔액이 0이면 반제 완결, 잔액이 남으면 미결 — 이 정도로 실무 파악 가능.

#### 나중에 필요하면

거래처별 잔액에서 특정 거래처의 잔액이 왜 남았는지 원장 드릴다운으로 확인 가능.  
실제로 라인 매칭이 필요해지면 그때 `clearings` 테이블 추가.

## 부가세 신고용 엑셀 다운로드 (갑지 + 명세 + 증빙 SQL)

**결정일:** 2026-07-16
**구현 범위:** `/vat` 페이지 + `GET /api/vat/export?start&end` (exceljs로 3개 시트 생성)

### 무엇을 만들었나

- **시트1 갑지:** 매출/매입 공급가액·세액·건수 합계, 납부(환급) 세액, 환급 계좌 입력란
- **시트2 명세:** 구분(매출/매입) | 거래일자 | 전표번호 | 적요 | 거래처명 | 계정과목 | 공급가액 | 부가세 | 합계
- 추출 규칙: `부가세예수금`(매출) / `부가세대급금`(매입) 라인이 있는 전표에서, **부가세 라인과 같은 방향**(매출=대변, 매입=차변)의 상대 라인(현금·부가세 계정 제외)을 공급가액으로 집계
  - 처음엔 상대 라인 전체를 차대 상계(net)했으나, 선급금·미지급금으로 정산하는 전표(예: #243 철거공사)에서 반대편 정산 라인이 공급가액을 깎아 음수가 됨 → 같은 방향 라인만 집계로 수정. 수정 후 전 행 세액 = 공급가액의 10% 확인
  - 부가세 납부/환급 전표(부가세 계정 + 보통예금만)는 상대 라인이 없어 자동 제외
  - 취소전표(`is_cancelled`) 제외, 부가세 계정이 없는 면세·해외결제는 자연 제외

### 무엇을 안 만들었나 / 왜

- **증빙종류(세금계산서/카드/현금영수증 구분) 컬럼:** DB에 해당 데이터가 없음. 대신 계정과목 컬럼으로 대체. 필요해지면 `journal_lines`에 증빙 유형 컬럼 추가 후 반영
- **홈택스 신고서식(정확한 갑지 양식) 재현:** 내부 증빙·세무사 전달용 요약표로 충분하다고 판단
- **"증빙 SQL" 시트 (처음엔 만들었다가 제거):** 추출 로직과 동등한 Raw SQL을 3번째 시트에 텍스트로 넣었었음. 하지만 국세청·세무대리인은 회사 내부 DB에 접근권이 없어 SQL을 실행할 방법이 없고, 그들이 요구하는 건 세금계산서 등 증빙 문서지 쿼리문이 아님 — "세무대리인/국세청 소명용"이라는 문구 자체가 과장이었다고 판단해 삭제. SQL이 필요해지면(내부 감사 추적용) 코드 주석이나 별도 문서로 남기는 게 맞음
